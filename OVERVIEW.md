# 쉽게 이해하는 Multi AI Agent 시스템

> 자동차 부품 공장에서 AI 에이전트 6명이 팀을 이뤄 일하는 시스템입니다.

---

## 한 줄 요약

**"공장의 재고를 최소한으로 유지하면서도, 고객 납기를 절대 못 맞추는 일이 없도록 AI들이 알아서 협력."**

---

## 비유로 이해하기

일반 공장은 사람이 모니터를 보고 판단합니다.
이 시스템은 **6명의 AI 전문가**가 각자 맡은 분야를 24시간 감시하면서, 문제가 생기면 **회의(CNP)**를 열어 함께 해결합니다.

```
공장 라인

  [프레스] → [용접] → [검사] → [창고] → [출하]
     EA        QA       QA       IA       DA
   설비전문가  품질전문가       재고전문가  수요전문가

                    PA (팀장)
                   모두를 조율
```

## 6명의 AI 에이전트

### EA — 설비 전문가
- **하는 일**: 프레스 기계의 진동, 온도를 실시간 감시
- **판단 예시**: "진동이 3.5mm/s를 넘었다 → 기계가 마모되고 있다 → 팀장에게 보고"
- **도구**: 예지보전 AI, 가용능력 예측 AI

### QA — 품질 전문가
- **하는 일**: 만들어진 부품의 두께, 버(burr), 평탄도 측정
- **판단 예시**: "Cpk가 1.0 아래로 떨어졌다 → 불량이 늘어날 것 → 팀장에게 보고"
- **도구**: 비전검사 AI, 불량예측 AI, SPC 분석

### SA — 공급 전문가
- **하는 일**: 강판, 용접 와이어 등 자재 재고와 AGV(무인운반차) 상태 감시
- **판단 예시**: "강판이 3시간치밖에 안 남았다 → 긴급 보충 요청"
- **도구**: 자재 버퍼 분석

### DA — 수요 전문가
- **하는 일**: 고객 주문량 변화와 수요 예측 정확도 추적
- **판단 예시**: "현대자동차에서 긴급 주문 +100개 → 수요 변동성 높아짐 → 팀장에게 보고"
- **도구**: 수요예측 AI (ARIMA/Prophet mock)

### IA — 재고 전문가
- **하는 일**: 완제품 창고의 재고량과 안전재고(SS) 모니터링
- **판단 예시**: "재고 45개인데 안전재고가 50개 → 위험! → 팀장에게 보고"
- **도구**: 안전재고 공식 계산, 리드타임 추정

### PA — 팀장 (계획 에이전트)
- **하는 일**: 다른 5명의 보고를 모아서 종합 판단, 회의(CNP) 주관
- **판단 예시**: "설비 경고 + 품질 저하가 동시에 발생 → 전체 회의 소집"
- **도구**: LLM (GPT-4o), 규칙 엔진

## 핵심 흐름: 문제 발생 → 해결까지

### 평소 (정상 상태)

```
매 택트(기본 약 2초, `MAS_TAKT_SEC`)마다:
  공장 → 부품 1개 생산 → 검사 → 양품이면 창고 입고 → 주문 있으면 출하

동시에:
  EA: "진동 2.1mm/s... 정상" (2초마다 체크)
  QA: "Cpk 1.45... 양호" (3초마다 체크)
  SA: "강판 850장... 충분" (5초마다 체크)
  DA: "수요 안정적" (4초마다 체크)
  IA: "재고 120개 > SS 31개... OK" (3초마다 체크)
  PA: "보고 없음... 대기" (2초마다 체크)
```

### 문제 발생!

```
진동이 서서히 올라간다... 3.5 → 3.8 → 4.1 mm/s

EA: "[경고] 위험! 진동 4.1mm/s → PA에게 알림 전송"
QA: "[경고] Cpk 0.85로 떨어짐 → PA에게 알림 전송"

PA: "경보 2개 수신! → 전체 회의(CNP) 소집!"
```

### 회의 (Contract Net Protocol)

```
PA → 전원: "상황이 심각합니다. 각자 대안을 제시해주세요" (CFP)

EA: "제안: 속도 80%로 감속하면 진동이 안정됩니다" (PROPOSE)
QA: "제안: 전수검사로 전환하면 불량 유출 방지 가능" (PROPOSE)
SA: "제안: 물류 루트 조정해서 자재 버퍼 확대" (PROPOSE)
DA: "제안: 고객에게 48시간 납기 연장 협상" (PROPOSE)
IA: "제안: 안전재고를 낮춰도 서비스레벨 유지 가능" (PROPOSE)

PA: 5개 제안을 종합 평가 (품질 30%, 납기 25%, 비용 25%, 안전 20%)
    → GPT-4o에게 복잡한 판단 위임 (또는 규칙 기반 자동 판단)
    → 통합 전략 확정: "속도 80%, 전수검사, 모니터링 10초 주기"

PA → 전원: "이 전략대로 실행하세요" (ACCEPT)

EA: PLC에 감속 명령 전송 [완료]
QA: 전수검사 모드 전환 [완료]
SA: AGV 경로 재배치 [완료]
IA: 안전재고 31개 → 12개로 하향 [완료]
```

### 결과

```
- 진동 안정화 (4.1 → 2.4 mm/s)
- 불량 유출 방지
- 안전재고 31 → 12로 줄어듦 = 재고 비용 절감!
- 서비스레벨 100% 유지
```

## 판단의 두 가지 방법

이 시스템은 **빠른 판단**과 **깊은 판단**을 구분합니다.

| | 규칙 기반 (Rule) | AI 기반 (LLM) |
|---|---|---|
| **속도** | < 1ms (즉시) | 1~3초 |
| **용도** | 안전, 긴급 상황 | 복잡한 전략 판단 |
| **예시** | "진동 5.5 이상 → 즉시 정지" | "설비+품질+수요 동시 문제 → 최적 대안은?" |
| **실패 시** | — | 규칙 기반으로 자동 전환 |

```
문제 발생
  │
  ├─ 안전 관련? ──→ 즉시 규칙 적용 (정지/감속)
  │
  ├─ 단순 임계값? ──→ 규칙 적용 (알람/경고)
  │
  └─ 복잡한 상황? ──→ GPT-4o에게 분석 위임
       (2개 이상 에이전트     └─ 실패 시 → 규칙 폴백
        동시 경보)
```

## 통신 구조

에이전트들이 대화하는 방법은 3가지입니다.

### 1. 메시지 브로커 (에이전트 ↔ 에이전트)
```
EA ──"설비 경고"──→ [브로커: alerts 토픽] ──→ PA의 인박스
```
- 9개 주제별 채널 (설비, 품질, 공급, 수요, 재고, 계획, CNP, 알람, 전체)
- 메시지가 반드시 전달되도록 보장 (재시도 + 확인)

### 2. MQTT (센서 → 에이전트)
```
프레스 센서 ──→ [MQTT: factory/line3/press-01/vibration] ──→ EA
```
- 실제 공장의 센서 통신 프로토콜과 동일
- Mosquitto 미설치 시 자동으로 내부 모드 전환

### 3. REST API (외부 → 시스템)
```
브라우저 ──→ http://localhost:8787 ──→ 실시간 대시보드
```

## 시나리오 테스트 (신규!)

이제 코드를 수정하지 않고도, **YAML 파일 하나만 바꿔서** 다양한 상황을 테스트할 수 있습니다.

### 준비된 시나리오 6종

| 시나리오 | 상황 | 목적 |
|----------|------|------|
| 정상 운영 | 모든 게 순조로움 | 기준선 (다른 시나리오와 비교용) |
| 설비 고장 | 프레스 기계가 빠르게 열화됨 | EA의 예지보전 능력 테스트 |
| 수요 폭증 | 긴급 주문이 60% 비율로 쏟아짐 | IA/PA의 재고 대응력 테스트 |
| 품질 위기 | 센서 노이즈 3배, 용접 편차 급증 | QA의 SPC/검사 전환 능력 테스트 |
| 자재 부족 | 강판 재고 1/3, 보충도 느림 | SA의 자재 관리 능력 테스트 |
| 복합 위기 | 위 4가지가 동시에 발생! | 전체 시스템의 종합 대응력 테스트 |

### 사용법

```bash
# 어떤 시나리오가 있는지 확인
python run_scenario.py --list

# 시나리오 실행 (100사이클 후 자동 종료, 결과 JSON 저장)
python run_scenario.py --scenario scenarios/normal.yaml --cycles 100
python run_scenario.py --scenario scenarios/equipment_failure.yaml --cycles 100

# 두 시나리오 결과를 나란히 비교
python compare_results.py results/normal_*.json results/equipment_failure_*.json
```

### 비교 결과 예시

```
══════════════════════════════════════════════════════════════
  시나리오 결과 비교 (2건)
══════════════════════════════════════════════════════════════
  KPI              │ 정상 운영         │ 설비 고장          │
  ─────────────────────────────────────────────────────────────
  총 사이클          │            30    │            30     │
  총 이벤트          │             2    │             4     │
  CNP 실행          │             2    │             2     │
  최종 재고          │            29    │            41     │
  안전재고           │             9    │            28     │
  서비스레벨          │        100.0%   │        100.0%    │
  AI 도구 호출       │            63    │            25     │
  Rule 판단          │             0    │            74     │
  ─────────────────────────────────────────────────────────────
```

설비 고장 시나리오에서는:
- **Rule 판단이 74회** 발생 → 설비 임계값 초과로 Decision Router가 자주 가동
- **안전재고가 28로 유지** → 불확실성이 커서 SS를 많이 낮추지 못함
- **AI 도구 호출이 25회** → 정상(63회)보다 적음 (감속으로 사이클이 줄어서)

### 나만의 시나리오 만들기

`scenarios/` 폴더에 새로운 YAML 파일을 만들면 됩니다:

```yaml
name: "나의 커스텀 시나리오"
description: "특정 상황을 테스트"

sensors:
  press:
    vibration:
      baseline: 3.0      # 초기 진동값을 높게
      noise_std: 0.5      # 노이즈도 크게
      drift_rate: 0.20    # 열화도 빠르게

warehouse:
  stock: 50               # 초기 재고 적게
  safety_stock: 40        # 안전재고 높게

events:
  vibration_spike: 0.30   # 진동 스파이크 30% 확률
  new_order: 0.25         # 주문 발생 25% 확률

execution:
  max_cycles: 200         # 200사이클 실행
```

## 안전재고 공식

이 시스템의 핵심 목표는 이 숫자를 최소화하는 것입니다.

```
안전재고 = z × √(리드타임 × 수요변동² + 평균수요² × 리드타임변동²)
```

쉽게 말하면:
- **수요가 예측 가능하면** → 안전재고를 줄일 수 있음
- **기계가 안정적이면** → 리드타임 변동이 줄어 → 안전재고를 줄일 수 있음
- **품질이 좋으면** → 수율이 높아 → 안전재고를 줄일 수 있음

그래서 **6개 에이전트가 협력**해서 이 모든 불확실성을 줄이면,
안전재고를 **동적으로 낮출 수** 있는 것입니다.
